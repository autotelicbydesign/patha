# v0.10.1 — package as `patha-memory`; regex false-positive filters; language audit

A focused follow-up to v0.10.0. No new architectural surface; everything below is post-tag polish that earned its way into a patch release.

## What changed

### 1. The PyPI distribution name is `patha-memory`

`patha` was already taken on PyPI, so we ship as `patha-memory`. The Python import name stays `patha`. README, `Makefile`, and `docs/publish-to-pypi.md` all updated to the correct distribution name. Wheel metadata verified:

```
Name:             patha-memory
Version:          0.10.1
Requires-Python:  >=3.11
```

If you previously pip-installed `patha`, switch to:

```bash
pip install patha-memory
```

### 2. Three regex extractor false-positive filters

The default (zero-deps) regex extractor used to hit on three predictable false-positive shapes in real conversational text. All three now suppressed before single-value extraction runs:

- **Range filter** — `$100 to $500`, `$50-$200`, `$80–$120` (en-dash). Both ends of a price band were being extracted as separate purchases. Now: zero tuples.
- **Hypothetical filter** — `thinking about a $300 helmet`, `would cost $X`, `considering`, `if I bought`, `might buy`. A currency match within 50 chars of any of these markers now gets dropped.
- **Negated-purchase filter** — `didn't buy the $X`, `couldn't afford`, `returned the $X helmet`, `decided against`, `cancelled`. Same proximity treatment.

12 new test cases in `tests/belief/test_ganita.py::TestFalsePositiveFilters`.

### 3. "synthesis bypasses Phase 1" → "synthesis answer is independent of Phase 1's top-K"

The original phrasing overstated the implementation. Phase 1 still runs in parallel to populate retrieval context; only the synthesis *answer source* is independent of Phase 1's top-K. Audited and corrected across:

- `README.md` (receipt paragraph and power-user section)
- `docs/innovations.md`
- `CHANGELOG.md`
- `src/patha/__init__.py` (`Memory.recall` docstring)
- `examples/three_innovations_demo.py`
- `tests/belief/test_innovations_compose.py`
- `writeups/update_08_compression.md`

### 4. Test rename for clarity

`test_synthesis_intent_bypasses_phase1` → `test_synthesis_intent_independent_of_phase1`. Same test, more honest name.

### 5. Receipt paragraph restructured

The v0.10 receipt in README and `writeups/update_08_compression.md` is now a bulleted block with:

- Exact (not `~`) token numbers: `118,761 → 18,384` (6.5× reduction)
- A `¹ one question excluded` footnote on the KU 77/77 number
- Option-A wording on the extraction-quality claim: `~84% of failures are synthesis-bounded (per eval/multisession_diagnosis.py); full benchmark with stronger extractors is future work` — measured rather than overclaimed.

### 6. Pre-flight install verification

New: `scripts/verify_install.sh <wheel-or-pip-spec>` — creates a throwaway Python 3.11 venv, installs the wheel, runs a representative smoke test (basic `Memory.remember`/`recall` + the synthesis-intent path with all three filters). Exits non-zero on any failure.

## Still unmeasured at scale

Single-question tests with hybrid+local-LLM extraction recover synthesis-bounded answers that the regex extractor misses, but full-benchmark re-runs with stronger extractors (`qwen2.5:14b`, hosted LLMs) are not yet measured. A future v0.11 release will report verified extractor lift if the empirical work supports the headline.
