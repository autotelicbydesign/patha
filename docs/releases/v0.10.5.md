# v0.10.5 — first PyPI release

The first version published to real PyPI. v0.10.2, v0.10.3, and v0.10.4 were all uploaded to TestPyPI but never to real PyPI; review of the rendered TestPyPI pages drove iterative metadata + framing fixes before the first real-PyPI publish.

## What changed vs v0.10.4

- **Articulation Bridge** — renamed from *Articulation Layer*. Phase 3 isn't actually a runtime layer of Patha: `Memory.recall()` only executes the Retrieval Layer and the Belief Layer. The third piece is the connection between Patha's output and a user's LLM — in production it lives in the application code that calls `memory.recall()` and then prompts an LLM; in evaluation, our offline harness in `eval/run_answer_eval.py` exercises that same shape against benchmark JSON to measure how well the bridge produces correct answers.
- **README framing** — "The three layers" → "Two layers and one bridge." The asymmetry is now explicit: only two things execute on `.recall()`; the bridge runs in your application code.

No other code or framing changes vs v0.10.4.

## What changed vs v0.10.3

- **Articulation Layer** — was *Answer Layer (Upamāna)* in v0.10.3. *Articulation* reads more naturally in English than *Answer*. The obvious Sanskrit pairing — *śabda* — was rejected: in Mīmāṃsā, śabda is *infallible Vedic testimony as the source of authoritative knowledge of unseen realities*, whereas Phase 3 *measures* the LLM's articulated output *against* an external gold (testimony as candidate, not authority — opposite directionality). Pramāṇa-naming for the third piece is left open until a tradition fits cleanly. Renamed again in v0.10.5 to "Articulation Bridge" to capture that it's not a runtime layer.

## What changed vs v0.10.2

- **Author metadata** — `authors = "Stefi P. Krishnan"` (was `"stefi"`). The wheel records the author and the rendered PyPI page shows it; bumping the version is the only clean way to fix this once a wheel has been built.

  - **Retrieval Layer (Pratyakṣa)** — direct perception / lookup.
  - **Belief Layer (Anumāna)** — inference / belief evolution.

  Internal engineering filenames (`docs/phase_2_*`, `docs/phase_3_plan.md`) keep "Phase N" for release-history bookkeeping.

Replaces with a Vedic-recitation + Aboriginal-songline epistemology framing that surfaces the architectural claim above the fold.
- **Headline numbers above Quickstart** — R@5 = 1.000 on LongMemEval-KU, 6.5× token reduction, zero LLM tokens at recall, now visible in the first 200 words.
- **"Why Patha (vs Claude's built-in memory…)"** → **"Beyond default AI memory: where Patha fits in your stack"**. Same content, neutral framing.

No code or test changes vs the v0.10.2 git tag content; this is a metadata + README release.

## Architectural distinction (unchanged from v0.10.2)

`Memory.recall()` routes by question intent:

- **Retrieval** (*pratyakṣa*) — Retrieval Layer (7-view dense + BM25 + RRF + cross-encoder reranker + songlines) → Belief Layer (current-state filter) → direct-answer or structured summary.
- **Synthesis** (*anumāna*) — gaṇita queries the preserved tuple index exhaustively. **Zero LLM tokens at recall.** The Retrieval Layer still runs in parallel to populate retrieval context, but the synthesis answer is independent of its top-K.

Verified by `tests/belief/test_innovations_compose.py::test_synthesis_intent_independent_of_phase1`, which forces the Retrieval Layer to return `[]` and the gaṇita layer still recovers the canonical $185.

## What's in the wheel

- The PyPI distribution name is **`patha-memory`**. Python import name stays `patha`.
- Wheel metadata: `Name: patha-memory`, `Version: 0.10.5`, `Requires-Python: >=3.11`.
- Three regex extractor false-positive filters (range / hypothetical / negated-purchase) — `tests/belief/test_ganita.py::TestFalsePositiveFilters` covers them.
- Hybrid karaṇa extractor (`patha.belief.karana.HybridKaranaExtractor`) — regex enumerates every `$X`, LLM only labels semantically. Recall preserved; LLM cost paid once at ingest, never at recall.
- Filesystem-native ingest — `patha import obsidian-vault <path>`.
- Phase 3 (Articulation Bridge) scaffolding ships in `eval/answer_eval.py` + `eval/run_answer_eval.py`: `ClaudeLLM` / `OllamaLLM` / `NullTemplateLLM` adapters, five scorers (`exact_match`, `normalised_match`, `numeric_match`, `token_overlap_match`, `embedding_cosine_match`, `llm_judge_match`), a runner CLI, and a measured floor with NullTemplateLLM on KU 78q (5/78 = 6.4% — the bar a real LLM should beat).
