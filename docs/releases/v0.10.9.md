# v0.10.9 — MCP server hardening + `mcp` is now a core dependency

Triggered by a real "Server disconnected" failure in Claude Desktop. Diagnostic + audit + fix in one release.

## The bug Stefi hit

```
MCP patha: Server disconnected.
```

```
$ uv run patha-mcp
Traceback (most recent call last):
  File "/Users/stefi/.local/bin/patha-mcp", line 4, in <module>
    from patha.mcp_server import main
  File ".../patha/mcp_server.py", line 56, in <module>
    from mcp.server.fastmcp import FastMCP
ModuleNotFoundError: No module named 'mcp'
```

`uv tool install patha-memory` installs core dependencies only. `mcp` was in `[project.optional-dependencies].mcp` — so it got skipped. The `patha-mcp` binary existed, but its first import crashed.

**Fix:** `mcp>=1.0` moved into core `dependencies`. The `[mcp]` extra stays as an empty placeholder for backwards-compat, so `pip install 'patha-memory[mcp]'` still works for anyone with that pin in their docs.

## What else changed (mcp-builder skill audit)

While in there, audited `src/patha/mcp_server.py` against the MCP Python best-practices checklist and shipped fixes for every gap:

### Tool definitions

| Before | After |
|---|---|
| Tools take raw `str \| None` parameters with no validation | Pydantic `BaseModel`s with `Field(...)` constraints (length, descriptions, validators) |
| Tools have no `annotations` | Every tool sets `readOnlyHint`, `destructiveHint`, `idempotentHint`, `openWorldHint` |
| Tool bodies don't catch exceptions → any error crashes JSON-RPC frame → "Server disconnected" | Every tool wrapped in `try/except`, returns `{"error": {"code", "message", "details"}}` instead |
| `datetime.fromisoformat(asserted_at)` raises `ValueError` on malformed input → kills tool | Safe `_parse_iso_or_none` helper, returns `{"error": {"code": "invalid_timestamp"}}` |

### Pagination

`patha_history` previously returned every match in one response. On a large store this could blow context. Now:

```json
{
  "term": "...",
  "total": 1234,
  "count": 50,
  "offset": 0,
  "next_offset": 50,
  "has_more": true,
  "matches": [...]
}
```

`limit` defaults to 50, capped at 200. `offset` and `next_offset` paginate cleanly.

### Hard input limits

- `MAX_PROPOSITION_CHARS = 4000` (one proposition is one fact, not a document)
- `MAX_QUESTION_CHARS = 2000`
- `MAX_TERM_CHARS = 200`
- `MAX_CONTEXT_CHARS = 200`
- `MAX_HISTORY_LIMIT = 200`

These are enforced at the Pydantic layer, before the tool body runs.

### Code quality

- Inline-in-tool imports (`GanitaIndex`, `answer_aggregation_question`) moved to module top.
- Bare `except Exception` replaced with logged warnings using `type(e).__name__: {e}`. All log lines go to **stderr** — the no-print-to-stdout rule is documented in the module docstring.
- Server name aligned: `FastMCP("patha_mcp")` now follows the Python convention `{service}_mcp` (was `"patha"`).

### Instruction prompt

Adds one new rule for clients:

> If a tool returns an `{"error": ...}` object, the server is still alive — just relay the error message to the user and continue.

This means transient tool failures (a malformed timestamp, an oversize proposition, a transient extractor crash) no longer break the whole conversation.

## What did NOT change

- The MCP protocol contract for the four tools is unchanged in shape — argument names and semantics match v0.10.8.
- 24 MCP tests still pass against the refactored server.
- No architectural changes to the Retrieval Layer, Belief Layer, or Articulation Bridge.
- No detector changes, no benchmark deltas.

## Verifying the fix

Fresh py3.11 venv, install from PyPI, check `patha-mcp` imports:

```bash
python3.11 -m venv /tmp/patha-test && source /tmp/patha-test/bin/activate
pip install patha-memory==0.10.9
python -c "from patha.mcp_server import main, mcp; print('mcp.name:', mcp.name)"
# → mcp.name: patha_mcp
which patha-mcp
# → /tmp/patha-test/bin/patha-mcp
```

`scripts/verify_install.sh dist/patha_memory-0.10.9-py3-none-any.whl` passes the existing belief / synthesis smoke tests.

## For users on v0.10.7 / v0.10.8 with broken MCP

```bash
uv tool uninstall patha-memory
uv tool install patha-memory==0.10.9
patha install-mcp --install-detector full-stack-v8 -y
# Quit Claude Desktop fully (Cmd+Q) and reopen.
```

If you'd already wired `[mcp]` into your config (`pip install 'patha-memory[mcp]'`, `uvx --from 'patha-memory[mcp]' patha-mcp`) — that still works in v0.10.9. The extra is an empty placeholder; the deps it used to pin are now in core.
