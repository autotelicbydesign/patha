# v0.10.7 — metric relabel + first real-LLM Articulation Bridge measurement

A correctness pass on how we describe Patha's measured numbers, plus the first end-to-end-through-an-LLM number for the Articulation Bridge.

## What changed vs v0.10.6

### The relabel

In v0.10.5 and v0.10.6 the README led with **"End-to-end accuracy on KU: 1.000 (77/77)."** This was a category error: the metric scores whether the gold answer (or one of its synonyms) appears as a *substring* in Patha's emitted summary text. **No LLM is involved in scoring.** It measures what Patha *surfaces*, not what an LLM *does with it*.

**Relabelled to "Belief-Layer answer-recall"** across the README, `docs/benchmarks.md`, and the related claims:

- "End-to-end accuracy, KU 78q: 1.000 (77/77)" → **"Belief-Layer answer-recall, KU 78q: 1.000 (77/77)"**
- "End-to-end accuracy, full 500q LongMemEval-S: 0.952 (472/496)" → **"`patha.Memory` answer-recall, full 500q: 0.952 (472/496)"**
- "Multi-session 500q stratum (accuracy): 0.857 (114/133)" → **"Multi-session 500q stratum (answer-recall): 0.857 (114/133)"**
- "Claim B: Unified Patha end-to-end" → **"Claim B: Unified Patha answer-recall"**

A "three different metrics" block was added to the README explainer:

- **Retrieval R@5** — *did Phase 1 surface the gold session in the top-5?* (Retrieval Layer quality)
- **Belief-Layer answer-recall** — *does the gold answer appear as a substring in Patha's summary?* (Belief-Layer surface quality; previously mis-labelled "end-to-end")
- **Articulation Bridge end-to-end** — *given Patha's output as context, does the user's LLM produce an answer that matches the gold under a chosen scorer?* (the actual end-to-end answer accuracy with a real LLM in the loop)

The architecture and the underlying numbers haven't changed. The labels did.

### First real-LLM Articulation Bridge measurement

The Articulation Bridge scaffolding has shipped since v0.10.0 (engine, three LLM adapters, six scorers, runner CLI). v0.10.6's only published measurement was the **NullTemplateLLM floor** (5/78 = 0.064 on KU 78q with the numeric scorer), which is a deterministic stub that doesn't actually answer questions.

v0.10.7 ships **the first measurement with a real LLM in the loop**: `qwen2.5:14b-instruct` via local Ollama (Q4_K_M, ~9 GB, GPU-resident).

| Scorer | Accuracy on KU 78q |
|---|:---:|
| numeric (5% tol) | 12/78 = 0.154 |
| normalised_match | 2/78 = 0.026 |
| **token-overlap ≥0.6 (LongMemEval-S official scorer)** | **24/78 = 0.308** |
| token-overlap ≥0.4 | 34/78 = 0.436 |
| embedding-cosine ≥0.85 | 6/78 = 0.077 |
| embedding-cosine ≥0.55 | 36/78 = 0.462 |

Per-strategy on the canonical token-overlap run: ganita 5/41 (0.122), structured 19/37 (0.514). Wall time 12 minutes (10 s/question on warm GPU).

**What this number means.**

- The Articulation Bridge measurement framework runs end-to-end on a real LLM, exits cleanly, and produces a number above the NullTemplateLLM floor. The plumbing works.
- 0.308 with a 14B local quantized model is a **floor for "real LLMs in the loop"** — not a ceiling. Inspection of failures shows two clusters:
  1. **Hallucinated specifics** despite memory containing the correct value — e.g. gold "$400,000" → answer "$350,000."
  2. **Prose verbosity** that defeats strict scorers despite a correct fact — e.g. gold "Three times a week" → answer correctly mentions "2-3 times" inside a longer paragraph.
- A frontier-class model (Claude Sonnet 4, GPT-4o, Gemini 2.5 Pro) would likely reduce both clusters substantially. **Frontier-LLM measurement pending; will publish in v0.11.**

### Other

- `OllamaLLM.timeout_s` default bumped 60 → 240 seconds. The 60s default was too short for a 14B model on long prompts; the first eval run failed with a urllib socket timeout. 240s is generous enough for any reasonable local-model eval.

No code, test, or architecture changes vs v0.10.6 beyond the timeout bump and the metric relabel.

## Why this is shipped as v0.10.7 and not v0.11.0

The metric relabel is a correctness fix: the public metadata (the README on PyPI) was carrying a misleading label. v0.11 will be the proper milestone bump for the frontier-LLM measurement + multi-session strong-extractor run + BeliefEval-through-Bridge adapter + demo video. v0.10.7 is the integrity patch that goes out before more readers arrive.
